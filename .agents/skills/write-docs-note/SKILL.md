# 编写文档笔记

## 概述

用于在 `docs/` 下编写技术笔记的工作流。

目标不是“把字写满”，而是写出：

- 有开头：先说清楚这篇笔记解决什么问题
- 有主体：定义、机制、示例、边界讲明白
- 有收束：在合适的场景下给出总结、选择建议或落地建议
- 有真实感：说明适用场景、代价、限制，而不是只写优点

笔记默认使用中文，技术术语保留英文。

---

## 核心原则

### 1. 先判断笔记类型，再决定结构

不要把所有笔记都写成同一个模板。先判断当前主题属于哪一类，再组织章节：

- A. 介绍 / 解释型（工具、框架、平台、概念、原理）
- B. 流程 / 教程型（操作步骤、实战过程）
- C. 速查 / 备忘型（命令、参数、代码片段）
- D. 排障 / 复盘型（报错处理、问题定位）

### 2. “完整”不等于“面面俱到”

完整的意思是：该讲的讲到位，而不是机械堆章节。

以下内容不应全局强制：

- `如何落地`
- `如何选择`
- `常见问题`
- `参考资源`
- `工作原理`
- 每章小结

这些内容只在“题目需要时”加入。

### 3. 优先解决读者的问题

开头先回答：

- 这是什么？
- 为什么要了解它？
- 不知道它，会卡在哪？

正文再回答：

- 它怎么工作？
- 什么时候该用？
- 什么时候不该用？
- 实际怎么做？

### 4. 不写成纯列表堆砌

允许用列表、表格、代码块，但关键章节之间要有过渡句，解释因果和上下文。

## 工作流

```text
读取目标文件 → 理解主题与已有结构 → 判断笔记类型 → 输出中文大纲 → 用户确认 → 一次性写作/分段写作 → 自检完整性
```

### 第一步：读取目标文件

读取用户指定的目标文件。

记录以下信息：

- 文件当前状态：空文件 / 骨架 / 半成品 / 已有较完整内容
- 已有章节标题
- 是否已有固定风格或编号方式

要求：

- 保留用户已有章节标题和有效内容
- 优先扩写，不要无故推翻重写
- 如果原文结构明显混乱，可以在大纲阶段提出重组建议

### 第二步：理解主题并判断类型

在写大纲前，先判断题目属于哪类笔记：

#### A. 介绍 / 解释型

适用于：

- 工具、框架、平台（Codex、Next.js、Webpack、Git 工具链）
- 概念、机制、原理（TypeScript 工具类型、HTTP、编译原理、某个抽象概念）

重点：

- 它是什么，解决什么问题
- 工作原理（框架 / 机制类必写；工具类按复杂度决定）
- 安装 / 配置 / 使用（工具类加；纯概念类跳过）
- 与相近概念的区别 / 使用边界（概念类加；工具类按需）
- 适用 / 不适用场景，常见坑

→ 对应模板：模板 A

#### B. 流程 / 教程型

适用于：

- Git 工作流、发布流程、接入流程
- 从 0 到 1 的配置过程

重点：

- 前置条件
- 操作步骤（每一步说明为什么）
- 常见错误，最终结果

→ 对应模板：模板 B

#### C. 速查 / 备忘型

适用于：

- curl 参数速查、Git 命令备忘、快捷键、代码片段留用

重点：

- 快速可用，分类清晰，示例直接，尽量少废话
- 不需要强加”如何落地”或大段总结

→ 对应模板：模板 C

#### D. 排障 / 复盘型

适用于：

- 某个报错处理、问题定位过程、环境兼容性问题

重点：

- 问题现象，根因，排查过程，最终解法
- 如何避免再次发生

→ 对应模板：模板 D

### 第三步：必要时澄清范围

满足以下任一条件时，才问一个问题：

- 主题涵盖 3 个以上独立子方向（如”前端性能优化”覆盖网络、渲染、代码等）
- 同时符合 2 种以上笔记类型（如既是工具介绍也是实战教程）

问法：

> “这篇笔记更侧重哪一块：介绍解释、操作流程、速查备忘，还是排障复盘？”

其他情况不要追问，直接进入第四步。

### 第四步：输出中文大纲

先给中文大纲，等待用户确认后再写正文。

要求：

- 大纲必须贴合当前笔记类型
- 不要机械套模板
- 章节名要直接、清楚，避免空泛标题

示例提问：

> “我先按这个结构来写。大纲是否符合预期？可以调整，或直接开始写。”

### 第五步：写正文

用户确认后再开始写。

规则：

- 正常情况下一次写完
- 如果预计正文超过 600 行，可以分 3 到 4 个章节一批次写
- 保持整篇语气一致
- 长文要有收束，短文不要为了形式硬凑结尾

### 第六步：写完后自检

写完后自行检查：

- 开头是否说明了问题背景
- 正文是否解释了关键机制
- 示例是否足够具体
- 是否写了边界、限制、代价
- 章节是否顺
- 结尾是否符合这类笔记的需要

### 第七步：处理用户追问与增补

如果用户在笔记写完后继续追问，例如：

- “再补一下 xxx”
- “这里能不能加一段 xxx”
- “顺便把 xxx 也写进去”
- “这个知识点应该放哪里”

不要默认把内容直接追加到文末。

正确做法：

1. 先判断新增内容属于：
   - 对已有章节的补充
   - 新的小节
   - 需要独立成章的重要主题
2. 优先插入到语义最接近的位置，保持阅读路径自然
3. 如果新增内容会打断当前结构，可以小范围重组相关章节
4. 只有当新增内容与现有结构关联很弱时，才放到文末“补充”或“附录”

判断原则：

- 属于基础概念、前置认知：尽量前置
- 属于原理细节：放到“工作原理”或对应机制章节
- 属于配置项、命令、示例：放到“基本使用”或“常用技巧”
- 属于部署、上线、性能、排障：放到对应专题章节，不要塞进概念章节
- 属于零散但有价值的信息：可放“补充说明”“常见问题”或“参考”

目标：

- 让补充后的笔记仍然像一篇完整文章
- 避免“正文写完了，后面再挂一串补丁”
- 用户连续追问多次后，结构仍然清晰

### 第八步：必要时允许局部重排，避免越补越乱

如果用户的追问属于高频关键点，或者连续几次追问已经暴露出原目录不够合理，可以主动重排相关章节。

允许重排的典型情况：

- 用户补充的是核心概念，但原文把它放得太后
- 用户连续追问的内容已经形成一个独立主题
- 原本的“常用技巧”章节膨胀过大，应该拆分
- 原文顺序导致理解门槛过高，前后依赖倒置
- 原文是边写边补形成的，出现明显“补丁感”

重排时的要求：

- 只重排与新增内容直接相关的章节，避免整篇大改
- 优先保持已有有效内容，不要为了重排而删掉信息
- 重排后要让目录更顺，而不是更复杂
- 变动超过 3 个章节时，先向用户说明将做局部结构调整，再执行；变动在 3 个章节以内可直接调整

判断标准：

- 这次追问补进去后，读者第一次阅读是否更顺
- 是否减少了来回跳读
- 是否让相关概念、示例、技巧出现在更合适的位置
- 是否把“补充内容”变成了“正文的一部分”

## 写作规则

### 结构规则（默认遵守）

- 语言使用中文，技术术语保留英文
- `##` 级标题可使用“一、二、三”编号，也可沿用原文风格
- `###` 一般不加编号，除非原文已使用
- 代码块必须带语言标记，如 `bash`、`ts`、`js`
- 表格用于命令速查、参数说明、方案对比、选择建议
- 注意事项优先使用 blockquote
- 图片引用：`![描述](/docs/文件名.png)`（图片存于 `static/docs/`）
- 不需要 frontmatter

### 内容规则（默认遵守）

#### 1. 先设问，再给答案

不要一上来就下定义。先说读者为什么要关心这个主题。

反例：

```markdown
❌ Vue Router 是 Vue.js 官方路由库。
```

正例：

```markdown
✅ 在多页应用里，每次跳转都刷新整个页面，URL 还不一定方便分享。
Vue Router 解决的就是这个问题：让前端页面切换不刷新，同时保留可访问、可分享的路由地址。
```

#### 2. “怎么用”和“为什么这样用”都要写

如果只写操作，不解释原因，读者很难迁移到别的场景。

复杂流程优先用编号步骤描述。

#### 3. 写优点时，也写限制和边界

只写优点会让笔记失真。以下内容至少命中其一：

- 不适合的场景
- 常见误区
- 使用代价
- 容易踩坑的地方

#### 4. 用具体例子，不要空话

优先使用：

- 真实命令
- 完整代码片段
- 具体输入输出
- 明确的场景描述

避免：

- “性能更好”
- “开发更方便”
- “比较灵活”

除非后面马上解释“具体好在哪”。

#### 5. 用段落把内容串起来

不要把一整章都写成散乱 bullet。

在这些地方补 1 到 2 句过渡：

- 从背景切到定义
- 从定义切到原理
- 从原理切到实战
- 从多个方案切到对比结论

#### 6. 结尾按需写，不做全局强制

以下几种收束方式，按主题选择其一即可：

- `总结`
- `如何选择`
- `实践建议`
- `如何落地`
- `避坑建议`

适用原则：

- 工具 / 流程 / 架构类主题：通常应该有结尾收束
- 速查 / 备忘 / 单点知识：可以没有结尾
- 多方案对比：建议写“如何选择”
- 面向实战：建议写“如何落地”或“实践建议”

#### 7. 来源可追溯：有引用必须标注

如果笔记内容来自用户提供的外部文章、URL 或研究报告，必须做来源归因，不能无声吸收后变成"自己的话"。

**触发条件**（满足任一）：

- 用户在请求时提供了参考 URL 或文章
- 笔记中引用了具体数据、研究结论、他人观点
- 内容主体基于某一特定作者的方法论或框架

**操作规范**：

- 在引用数据的表格或段落后添加脚注标记（如 `[^1]`）
- 在正文首段提及原始作者姓名（如有），并用脚注标注来源
- 在笔记末尾添加 `## 参考资料` 章节，列出完整引用：
  ```markdown
  [^1]: 作者名 — [文章标题](URL) · 来源平台 · 年份
  ```
- 引用多个来源时，每条独立编号

**不需要标注的情况**：

- 内容为笔记作者自己的实践总结，无外部来源
- 通用知识或官方文档中的基础定义（但官方文档链接仍建议放在参考资料中）

#### 7. 章节总结不是强制项

只有在以下情况才建议加“小结”：

- 长文
- 章节较多
- 原理链路复杂
- 需要帮助读者记忆重点

短文、速查文、API 单点说明不要硬加小结。

## 推荐大纲模板

以下模板是“参考骨架”，不是硬性套版。

### 模板 A：介绍 / 解释型

```markdown
## 引言（复杂主题时再写）

先说不用它 / 不理解它会遇到什么问题。

---

## 一、它是什么

核心定义 + 解决的问题。

---

## 二、工作原理（必要时）

用步骤或流程讲清楚机制。

---

## 三、安装与配置（工具类加，纯概念类跳过）

写清前置条件、命令和关键配置。

---

## 四、基本使用 / 示例

放最小可运行示例或典型用法。

---

## 五、与相近概念的区别 / 适用场景与限制（按需）

概念类：写区别和使用边界；工具类：写适合用、不适合用、常见坑。

---

## 六、实践建议 / 如何选择 / 总结（按需三选一）
```

### 模板 B：流程 / 教程型

```markdown
## 一、背景与目标

这套流程解决什么问题，最终要达成什么结果。

---

## 二、前置条件

环境、权限、依赖、输入材料。

---

## 三、操作步骤

按顺序写，每一步说明为什么。

---

## 四、常见问题或注意事项（按需）

---

## 五、总结 / 实践建议（按需）
```

### 模板 C：速查 / 备忘型

```markdown
## 一、用途说明（可选，1 到 3 句）

---

## 二、速查内容

用表格、命令、示例、分类组织。

---

## 三、补充说明 / 参考（按需）
```

### 模板 D：排障 / 复盘型

```markdown
## 一、问题现象

---

## 二、根因分析

---

## 三、解决过程

---

## 四、最终方案

---

## 五、如何避免再次发生（按需）
```

## 何时应该加哪些章节

### 建议加“工作原理”

适用于：

- 框架
- 中间件
- 编译 / 构建工具
- 协议 / 机制
- 涉及调用链的复杂主题

### 建议加“如何选择”

适用于：

- 同时介绍 2 种以上方案
- 工具有明显取舍
- 用户需要做技术决策

### 建议加“如何落地”或“实践建议”

适用于：

- 工具引入
- 团队流程
- 架构调整
- 工作流设计
- 读者看完后需要“下一步怎么做”

不适用于：

- 单一 API
- 工具类型
- 命令速查
- 简短备忘

### 建议加“常见问题”

适用于：

- 新手高频踩坑
- 安装配置易出错
- 实际操作路径较长

### 建议加“参考资源”

适用于：

- 主题较大
- 涉及官方文档
- 文中引用了外部资料

## 风格参考

### 好的写法特征

- 开头让读者知道“为什么要看”
- 正文有“定义 + 原理 + 示例 + 边界”
- 不回避缺点和限制
- 例子能直接运行或直接照做
- 长文有收束，短文不做作

### 要避免的写法

- 一上来就堆百科定义
- 全文只有 bullet，没有解释
- 大量命令，没有上下文
- 只有优点，没有代价
- 每篇都硬套“如何落地”
- 每一章都机械加“小结”

---

## 执行摘要

每次写笔记前，按上方工作流（第一步至第八步）顺序执行。局部补充时同样遵守类型判断和安放位置的原则，不要盲目追加到文末。
